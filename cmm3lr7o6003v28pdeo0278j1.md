## Pourquoi un script d'audit "maison" ?

On pourrait utiliser des outils d'audit industriels, mais coder le sien permet de comprendre ce qui se passe sous le capot.

## Sherlock Nexus est sur le coup !

## Le script : my-safe-host (V2)

Ce script est modulaire. Il vérifie les fichiers sensibles, ta config SSH, tes services réseaux et tes mises à jour.

```bash
#!/bin/bash

# ==============================================================================
# SCRIPT : my-safe-host.sh (V2.0)
# DESCRIPTION : Audit de sécurité avec gestion d'erreurs et détails processus.
# ==============================================================================

RED='\033[0;31m'
GREEN='\033[0;32m'
YELLOW='\033[1;33m'
BLUE='\033[0;34m'
NC='\033[0m'

echo -e "${BLUE}==================================================${NC}"
echo -e "${BLUE}       AUDIT DE SÉCURITÉ : $(hostname)           ${NC}"
echo -e "${BLUE}       DATE : $(date +'%d/%m/%Y %H:%M')          ${NC}"
echo -e "${BLUE}==================================================${NC}\n"

# --- 1. PERMISSIONS ---
echo -e "${YELLOW}[*] Vérification des fichiers sensibles...${NC}"
if [ -f /etc/shadow ]; then
    SHADOW_PERMS=$(stat -c "%a" /etc/shadow)
    if [ "$SHADOW_PERMS" -le "640" ]; then
        echo -e "  [${GREEN}OK${NC}] /etc/shadow est sécurisé ($SHADOW_PERMS)"
    else
        echo -e "  [${RED}!!${NC}] DANGER : /etc/shadow est trop ouvert ($SHADOW_PERMS)"
    fi
else
    echo -e "  [${YELLOW}??${NC}] /etc/shadow introuvable. Système non standard ?"
fi

# --- 2. SSH  ---
echo -e "\n${YELLOW}[*] Analyse de la configuration SSH...${NC}"
if [ -f /etc/ssh/sshd_config ]; then
    # Root Login
    ROOT_LOGIN=$(grep "^PermitRootLogin" /etc/ssh/sshd_config | awk '{print $2}')
    ROOT_LOGIN=${ROOT_LOGIN:-"non défini (défaut: yes)"} # Gère le cas où la ligne n'existe pas
    
    if [[ "$ROOT_LOGIN" == "no" || "$ROOT_LOGIN" == "prohibit-password" ]]; then
        echo -e "  [${GREEN}OK${NC}] Accès Root restreint ($ROOT_LOGIN)"
    else
        echo -e "  [${RED}!!${NC}] DANGER : PermitRootLogin est à $ROOT_LOGIN"
    fi
else
    echo -e "  [${GREEN}OK${NC}] SSH Server non détecté (Le port n'est pas exposé)."
fi

# --- 3. RÉSEAU (AVEC PROCESSUS) ---
echo -e "\n${YELLOW}[*] Services écoutant sur le réseau...${NC}"
# On ajoute '-p' pour voir le nom du programme et on filtre mieux
EXPOSED=$(ss -tulnp | grep -E '0.0.0.0|:::|127.0.0.1' | grep "LISTEN")

if [ -z "$EXPOSED" ]; then
    echo -e "  [${GREEN}OK${NC}] Aucun service en écoute."
else
    echo -e "  [${YELLOW}INFO${NC}] Services actifs (Vérifie si tu les connais) :"
    # On nettoie l'affichage pour que ce soit lisible
    echo "$EXPOSED" | awk '{print "      - Port: " $5 " | Programme: " $7}' | sed 's/users:(("//g;s/"))//g;s/",.*//g'
fi

# --- 4. MISES À JOUR ---
echo -e "\n${YELLOW}[*] Mises à jour...${NC}"
if command -v apt &> /dev/null; then
    UPDATES=$(apt list --upgradable 2>/dev/null | grep -ic "security")
    if [ "$UPDATES" -eq 0 ]; then
        echo -e "  [${GREEN}OK${NC}] Aucun patch de sécurité en attente."
    else
        echo -e "  [${RED}!!${NC}] $UPDATES patch(s) de sécurité critiques !"
    fi
fi

echo -e "\n${BLUE}==================================================${NC}"
```

### Décryptage du script : Comment fonctionne l'audit ?

Pour comprendre ce que fait Sherlock Nexus sous le capot, décomposons les quatre étapes clés du script.

#### 1\. La gestion du style (UI)

Au début, on définit des variables de couleurs (`RED`, `GREEN`, etc.). En Bash, on utilise des codes ANSI (ex: `\033[0;31m` pour le rouge).

*   **Pourquoi ?** Pour que l'œil de l'utilisateur soit immédiatement attiré par ce qui pose problème (`RED`) ou ce qui est validé (`GREEN`).
    

#### 2\. L'inspection des fichiers sensibles

```shell
SHADOW_PERMS=$(stat -c "%a" /etc/shadow)
```

Le script utilise la commande `stat -c "%a"`. Elle ne lit pas le contenu du fichier (trop risqué), elle regarde ses **droits d'accès**.

*   **Le test :** Si le score est supérieur à **640**, cela signifie que des utilisateurs non autorisés pourraient potentiellement lire les mots de passe hachés du système. C'est une faille critique.
    

#### 3\. Le scanner SSH intelligent

Ici, le script fait preuve de jugeote :

*   **Étape A :** Il vérifie si le fichier `/etc/ssh/sshd_config` existe. S'il n'existe pas, Sherlock en déduit que le serveur SSH n'est pas installé. C'est un **\[OK\]** automatique car ce qui n'est pas là ne peut pas être piraté !
    
*   **Étape B :** S'il existe, il utilise `grep` pour chercher la directive `PermitRootLogin`. Si elle est à "yes", n'importe qui peut tenter de se connecter en tant qu'administrateur (Root). On préfère un accès restreint.
    

#### 4\. Le radar réseau (L'écoute des ports)

C'est la partie la plus technique. Le script lance la commande `ss -tulnp`.

*   `t` : TCP
    
*   `u` : UDP
    
*   `l` : Listen (ceux qui attendent une connexion)
    
*   `n` : Numérique (pour voir "80" au lieu de "http")
    
*   `p` : Processus (pour savoir quel logiciel utilise le port)
    

Le script utilise ensuite `awk` et `sed` pour "nettoyer" la sortie brute de Linux, qui est souvent illisible, afin de t'afficher uniquement l'essentiel : **Port | Programme**.

#### 5\. Le check des mises à jour

```shell
apt list --upgradable | grep -ic "security"
```

Le script demande à ton gestionnaire de paquets (`apt`) s'il y a des mises à jour, puis il compte (`-c`) combien contiennent le mot "security". C'est le meilleur indicateur pour savoir si ta machine est exposée à des failles connues (CVE).

### Comment lancer l'audit ?

Une fois que tu as copié le code dans un fichier nommé `my-safe-host.sh`, n'oublie pas de lui donner les droits d'exécution.

```shell
# On rend le fichier exécutable
chmod +x my-safe-host.sh
```

Ensuite, lance-le avec `sudo` pour que Sherlock Nexus puisse inspecter les fichiers protégés.

> **Note :** Le passage par `sudo` est indispensable ici car, sans lui, le script ne pourra pas vérifier si ton fichier `/etc/shadow` est bien sécurisé (et il t'affichera une erreur de permission).

```shell
# On lance l'audit avec les privilèges administrateur
sudo ./my-safe-host.sh
```

## Retour d'expérience : Ce que le script m'a appris

Lors de mon premier lancement de ce script mais version V1 , le script n'était pas parfait vu qu'il me rendait une erreur de faux positif.

1.  **Le faux positif SSH :** Le script hurlait au danger car je n'avais pas de serveur SSH installé. J'ai donc ajouté une vérification : si le fichier n'existe pas, c'est que le service n'est pas là, donc ça va !
    
2.  **Les services "fantômes" :** En utilisant l'option `-p` avec `ss`, j'ai découvert que mon `docker-proxy` et `cupsd` écoutaient sur le réseau. Rien de grave, mais maintenant, je *les avais oublié*.
    

### Le Lexique du Détective (Commandes Bash)

Si tu es comme moi débutant en scripting, voici les outils que nous avons utilisés pour extraire les informations du système :

*   `grep` : C'est ton filtre. Il parcourt un fichier ou un texte pour ne garder que les lignes qui contiennent un mot précis (comme "security" ou "PermitRootLogin").
    
*   `awk` : Un outil surpuissant pour manipuler les colonnes. Dans notre script, il nous sert à dire : "Prends seulement le 5ème et le 7ème mot de la ligne". C’est grâce à lui qu'on extrait proprement le numéro du port.
    
*   `sed` : C'est ton "gommeur". Il nous permet de supprimer les caractères inutiles (comme les parenthèses ou les guillemets superflus) pour que l'affichage reste élégant dans ton terminal.
    
*   `stat` : Contrairement à `ls`, `stat` donne les détails techniques d'un fichier. On l'utilise ici pour obtenir les permissions en format numérique (ex: 640).
    
*   `ss` (Socket Statistics) : C'est le remplaçant moderne de la vieille commande `netstat`. Il sert à voir tout ce qui entre et sort de ta machine par le réseau.
    
*   `command -v` : Une astuce de scripteur pour vérifier si un programme (comme `apt`) est installé avant d'essayer de s'en servir. Cela évite que le script ne plante lamentablement sur une distribution non-Debian.
    

**"Pour aller plus loin..."**

Ce script est une base solide, mais la sécurité est une course sans fin. Voici ce que j'ai prévu d'ajouter dans la prochaine version (ou ce que tu peux essayer de coder de ton côté :) ) :

### 1\. Vérification du Pare-feu (UFW)

C'est la base, mais ton script actuel ne regarde pas si le pare-feu est actif.

*   **L'idée :** Vérifier si `ufw` ou `iptables` est en route.
    
*   **Le code :** `ufw status | grep "active"`
    

### 2\. Détection des "World-Writable" Files

Ce sont les fichiers que n'importe qui sur la machine peut modifier (un gros risque de sécurité).

*   **L'idée :** Chercher les fichiers avec des permissions trop larges (777).
    
*   **La commande :** `find /home -xdev -type f -perm -0002`
    

### 3\. Audit des utilisateurs avec UID 0

Normalement, seul **root** doit avoir l'UID 0. Si un autre utilisateur apparaît, c'est souvent signe d'une intrusion ou d'une grosse erreur de config.

*   **L'idée :** Scanner `/etc/passwd`.
    
*   **Le code :** `awk -F: '($3 == "0") { print $1 }' /etc/passwd`
    

### 4\. Vérification du temps d'inactivité (Auto-Logout)

Sur un serveur sécurisé, on veut que la session se ferme toute seule si on oublie de déco.

*   **L'idée :** Vérifier si la variable `TMOUT` est configurée.
    

### 5\. Export des résultats en Log ou JSON

Pour un usage plus "DevOps", ce serait top que le script génère un petit rapport.

*   **L'idée :** Ajouter une option pour dumper le résultat dans un fichier `audit_sadhill_date.log`.

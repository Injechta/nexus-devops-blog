L'observabilité n'est pas qu'une question d'outils, c'est une culture. Si tu veux progresser, tu dois pouvoir mesurer ce que tu déploies. Dans ce guide, nous allons construire un environnement professionnel, éphémère et automatisé.

### Pourquoi mesurer ? Le standard DORA

Avant de manipuler `kind` ou `helm`, il est essentiel de comprendre l'objectif. Dans le monde du DevOps, on se réfère souvent aux **indicateurs DORA** (*DevOps Research and Assessment*). Ce sont quatre métriques clés qui permettent de savoir si ton équipe est performante :

1.  **Deployment Frequency :** À quelle fréquence déploies-tu ton code ?
    
2.  **Lead Time for Changes :** Combien de temps faut-il pour qu'un commit arrive en production ?
    
3.  **Change Failure Rate :** Quel pourcentage de tes déploiements échoue ?
    
4.  **Failed Service Recovery Time (MTTR) :** Combien de temps mets-tu pour réparer un service tombé en panne ?
    

**Sans observabilité, ces mesures sont impossibles.**

Construire ce lab, c'est mettre en place le système nerveux qui te permettra, à terme, d'afficher ces quatre indicateurs sur ton dashboard. Tu ne fais plus seulement du monitoring : tu construis les fondations de ta performance technique.

## 1\. Création du cluster Kind

Pour commencer, nous créons un cluster Kubernetes local isolé. Nous nommons ce cluster `lab-observabilite` pour le retrouver facilement.

```plaintext
kind create cluster --name lab-observabilite
```

## 2\. Installation de la stack Prometheus/Grafana

Nous allons utiliser **Helm**, le gestionnaire de paquets de Kubernetes, pour installer tout le nécessaire en une fois.

**Étape 2.1 : Ajouter le dépôt officiel de Prometheus**

```plaintext
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
```

**Étape 2.2 : Mettre à jour la liste des paquets**

```plaintext
helm repo update
```

![](https://cdn.hashnode.com/uploads/covers/6989fc595065ae2aa69fc161/474ecd32-cfa4-4b68-9009-ee3a4b9beee5.png align="center")

**Étape 2.3 : Créer un espace de nom (namespace) dédié**

Il est crucial d'isoler les outils d'observabilité dans un espace de nom séparé.

```plaintext
kubectl create namespace monitoring
```

![](https://cdn.hashnode.com/uploads/covers/6989fc595065ae2aa69fc161/b3ea8a8d-087f-4a7c-a62b-ab290cbab388.png align="center")

**Étape 2.4 : Installer la stack de monitoring**

Cette commande installe Grafana, Prometheus et l'Alertmanager. L'option `--wait` est ici pour s'assurer que tout est opérationnel avant de reprendre la main.

```plaintext
helm install monitoring-stack prometheus-community/kube-prometheus-stack --namespace monitoring --wait
```

![](https://cdn.hashnode.com/uploads/covers/6989fc595065ae2aa69fc161/9bcaa5ac-30d6-4536-b141-e6eda2e44cd9.png align="center")

### Vérifier le statut des Pods (La vue immédiate)

C'est la commande la plus utile pour voir si tes conteneurs sont en train de démarrer ou s'ils sont bloqués , tu peux par exemple la lancer dans un autre terminal par exemple si tu trouves que la partie **Installer la stack de monitoring** prend trop de temps :

```plaintext
kubectl get pods --namespace monitoring
```

![](https://cdn.hashnode.com/uploads/covers/6989fc595065ae2aa69fc161/e74f39ac-8a8b-4291-912c-cce8047f92d9.png align="center")

Suivant ce que te renvoies cette commande

*   `Running` : Le pod est actif.
    
*   `Pending` : Kubernetes attend une ressource (ex: un nœud disponible).
    
*   `ContainerCreating` : L'image est en cours de téléchargement.
    
*   `CrashLoopBackOff` : Il y a un problème de configuration ou de ressource dans le pod.
    

## 3\. Déploiement "Pro" avec le pattern Sidecar

Pour monitorer Nginx, nous n'allons pas simplement le déployer. Nous allons ajouter un **exportateur de métriques** en tant que **Sidecar**. C'est le standard en entreprise : ton application sert les pages, et le conteneur `nginx-exporter` traduit les métriques pour Prometheus.

**Étape 3.1 : Création du déploiement**

Crée un fichier nommé `nginx-lab.yaml` avec ce contenu :

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-demo
  namespace: default
spec:
  replicas: 2
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:alpine
        ports:
        - containerPort: 80
      - name: nginx-exporter # Notre Sidecar
        image: nginx/nginx-prometheus-exporter:latest
        args:
          - -nginx.scrape-uri=http://localhost:80/stub_status
        ports:
        - containerPort: 9113
          name: http-metrics
---
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
  namespace: default
spec:
  ports:
  - name: web
    port: 80
    targetPort: 80
  - name: http-metrics # IMPORTANT: Le port du sidecar doit être déclaré ici
    port: 9113
    targetPort: 9113
  selector:
    app: nginx
```

**Étape 3.2 : Application de la configuration**

```shell
kubectl apply --filename nginx-lab.yaml
```

## 4\. Automatisation avec le ServiceMonitor

Pour que Prometheus "voit" automatiquement ton serveur Nginx, Il faut lui dire explicitement de regarder le port nommé `http-metrics`.

Crée un fichier nommé `nginx-monitor.yaml` :

```yaml
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: nginx-monitor
  namespace: default # Important : même namespace que le Service
  labels:
    release: monitoring-stack
spec:
  selector:
    matchLabels:
      app: nginx
  endpoints:
  - port: http-metrics # Cible le port déclaré dans le Service
    path: /metrics
    interval: 30s
```

Applique la règle :

```plaintext
kubectl apply --filename nginx-monitor.yaml
```

### 6\. Le guide de survie : Quand ça ne marche pas (Dépannage)

Dans Kubernetes, si un pod reste bloqué, ne panique pas. Voici tes deux réflexes de survie :

**Si ton pod est en** `CrashLoopBackOff` **ou** `ImagePullBackOff` **:**

Regarde les logs pour voir ce qu'il se passe à l'intérieur :

```plaintext
kubectl logs --namespace monitoring --selector app.kubernetes.io/name=grafana
```

*(Remplace* `grafana` *par le nom de l'application qui pose problème).*

![](https://cdn.hashnode.com/uploads/covers/6989fc595065ae2aa69fc161/715a65b7-377e-4a31-8c70-b872aa6cdb83.png align="center")

**Si tu veux savoir pourquoi un pod refuse de démarrer :**

La commande `describe` te donne les événements récents (les erreurs de réseau, de mémoire ou de droit) :

```plaintext
kubectl describe pod --namespace monitoring --selector app.kubernetes.io/name=grafana
```

* * *

La partie Events peut aider à déboguer quand quelque chose ne fonctionne pas, içi on peut voir que pour le moment tout va bien

![](https://cdn.hashnode.com/uploads/covers/6989fc595065ae2aa69fc161/1fbcf164-7c2c-440f-9b30-5926a9db346d.png align="center")

Pour valider que tout est vraiment OK maintenant, fais ceci :

**Vérifie l'état de tous les déploiements :**

```plaintext
kubectl get deployments --namespace monitoring
```

Tu devrais voir une colonne `READY` avec des chiffres comme `1/1` ou `2/2` pour chaque composant.

![](https://cdn.hashnode.com/uploads/covers/6989fc595065ae2aa69fc161/d3a7e899-7f50-4149-80a6-af82a645fd92.png align="center")

**Vérifie spécifiquement Prometheus :** Le pod de Prometheus est souvent le plus long à démarrer car il doit préparer sa base de données (TSDB). Regarde s'il est prêt :

```plaintext
kubectl get pods --namespace monitoring --selector app.kubernetes.io/name=prometheus

```

![](https://cdn.hashnode.com/uploads/covers/6989fc595065ae2aa69fc161/83d5f589-9529-43c6-b63f-a4f37c6e23e5.png align="center")

**Si tout est en** `1/1` **ou** `2/2` **:** Ton installation est terminée. Tu peux ignorer le fait que `helm` était "bloqué" (tu peux faire un `Ctrl+C` si la commande est toujours active dans ton terminal, Helm a déjà fini son travail de création des ressources Kubernetes).

> Si ton statut dans Prometheus affiche `DOWN` **(404 Not Found)** : Ne panique pas ! Ce 404 est une excellente nouvelle : cela signifie que Prometheus **a trouvé** ton service, mais que l'URL `/metrics` ne renvoie rien.
> 
> *   **Vérifie tes endpoints :** `kubectl get endpoints nginx-service`. Si tu vois tes IP, le réseau est OK.
>     
> *   **Vérifie ton port :** Est-ce que tu as bien déclaré le port `9113` dans ton `Service` ?
>     
> 
> *Reflexe de survie :* `kubectl logs <nom-du-pod>` (pour voir les erreurs de l'exporter) et `kubectl describe pod <nom-du-pod>` (pour les problèmes de démarrage).

### 7\. Nettoyage : Remise à zéro

Tu as terminé tes tests et tu veux repartir d'une page blanche ? Pas besoin de chercher chaque fichier un par un, Kubernetes s'occupe de tout.

**Étape 7.1 : Supprimer ton application Nginx** Comme tu as utilisé un fichier manifeste, supprime simplement toutes les ressources définies dedans :

```plaintext
kubectl delete -f nginx-lab.yaml
kubectl delete -f nginx-monitor.yaml
```

**Étape 7.2 : Supprimer la stack de monitoring** Désinstalle la stack proprement via Helm pour libérer les ressources :

```plaintext
helm uninstall monitoring-stack --namespace monitoring
kubectl delete namespace monitoring
```

**Étape 7.3 : Détruire le cluster** Pour supprimer définitivement le cluster Kind et libérer la mémoire de ta machine :

```plaintext
kind delete cluster --name lab-observabilite
```

*Résultat : ton ordinateur est comme neuf, aucune trace ne subsiste. On est propre et prêt pour la suite !*

### 8\. Audit de fin : S'assurer qu'il ne reste rien

Après avoir lancé tes commandes de suppression, il est important de vérifier que ton système est revenu à son état initial.

**Étape 8.1 : Vérifier la disparition des clusters** Vérifie qu'aucun cluster `kind` ne tourne encore sur ta machine :

```plaintext
kind get clusters
```

*Si la commande ne renvoie rien, c'est que ton cluster est bien supprimé.*

**Étape 8.2 : Vérifier les espaces de noms (namespaces)** Vérifie qu'il ne reste aucune trace de ton espace de monitoring :

```plaintext
kubectl get namespaces
```

*Si tu vois encore* `monitoring` *dans la liste, attends quelques secondes (Kubernetes a parfois besoin d'un temps de grâce pour purger les ressources).*

**Étape 8.3 : Supprimer le dossier de travail** Enfin, pour clore le chapitre de ton lab, supprime les fichiers que tu as créés sur ton ordinateur :

```plaintext
# Dans ton dossier ~/Documents/projets/tutos/monitoring_observabilite/
rm nginx-lab.yaml nginx-monitor.yaml
cd ..
rmdir monitoring_observabilite
```

## 9\. L'Observabilité : Bien plus qu'un simple graphique

Nous avons construit une base solide, mais l'observabilité est un voyage qui ne s'arrête jamais. Pour que ton cluster passe d'un simple "outil de monitoring" à une véritable **plateforme d'observabilité**, voici les deux piliers que nous explorerons ensemble dans le prochain épisode :

### Défi 1 : La Persistance des Données (Le stockage durable)

Dans notre lab actuel, si tu fais `kind delete cluster`, toutes tes données (historique des logs, métriques de Prometheus) disparaissent. En entreprise, on utilise des **PersistentVolumeClaims (PVC)**.

*   **Prochain défi :** Apprendre à monter un dossier de ton PC Linux (`sadhill`) directement dans le Pod Prometheus. Ainsi, même après la suppression du cluster, tes données resteront intactes.
    

### Défi 2 : L'Alerte intelligente

Monitorer, c'est bien. Savoir quand agir, c'est mieux.

*   **Prochain défi :** Configurer l'**Alertmanager** pour recevoir une notification sur ton terminal (ou via un webhook) dès que ton déploiement `nginx-demo` tombe à 0 réplique. On ne veut plus surveiller les courbes : on veut être prévenus quand le service est en danger.
    

### Défi 3 : La maîtrise du Dashboard Custom

Ne te contente plus des imports automatiques.

*   **Prochain défi :** Créer ton propre tableau de bord Grafana de A à Z. Nous apprendrons à manipuler le langage **PromQL** pour créer des graphiques spécifiques, comme ton propre compteur de taux d'erreur HTTP 404, afin de détecter les liens cassés sur ton site en temps réel.
